# Objective

Next.js page at `/admin/brightdata` with a trigger panel (city selector + trigger button + confirmation dialog) and a status display (last triggered snapshot id + current status with manual refresh). Confirmation dialog blocks accidental clicks (BrightData calls cost money). Manual poll only — no auto-refresh / SSE this sprint (per Out-of-Scope #3 in brief).

# Files

- `frontend/src/app/admin/brightdata/page.tsx` (new)
- `frontend/src/app/admin/brightdata/__tests__/page.test.tsx` (new)

# Acceptance Criteria

- [ ] `frontend/src/app/admin/brightdata/page.tsx`:
  - City selector (FW / Dallas / Montgomery — same set as resource page)
  - "Trigger Pre-Crawl" button → opens shadcn `AlertDialog` confirmation → on confirm, calls `triggerCrawl(payload)` from T26.7
  - Button disabled while in-flight (loading spinner)
  - Status panel below: shows last triggered snapshot id + current status (`running` / `done` / `failed`) + last refreshed timestamp
  - "Refresh Status" button → calls `getCrawlStatus(snapshotId)` (manual poll; no auto-refresh)
  - Last triggered snapshot persists in `localStorage` (so page-reload preserves the id; no backend session)
- [ ] `frontend/src/app/admin/brightdata/__tests__/page.test.tsx` covers:
  - Trigger button disabled while in-flight
  - Confirmation dialog blocks accidental clicks (close-without-confirm does not call API)
  - Trigger calls API with correct payload (city + any other required fields)
  - Status display renders all 3 status states (running/done/failed)
  - Refresh button polls API
  - localStorage persistence: write on trigger, read on mount
